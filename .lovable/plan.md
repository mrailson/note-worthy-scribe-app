

# Fix: Hallucinated Meeting Notes from Short/Test Recordings

## Problem

The meeting "Mon 23rd March 2026 (8:04 am)" has a 172-word, 4-minute test recording about insurance infrastructure. However, the generated notes contain entirely fabricated NHS content: "Anshal will finalise the Bugbrooke service testing protocols", "Railson assigned the PCN DES reporting task", "15% increase in ARRS recruitment targets" — none of which appear in the transcript.

## Root Cause

Two failures combine to produce this:

1. **No minimum word guard on the edge function.** The `generate-meeting-notes-claude` function accepts any transcript, no matter how short. When given a ~170-word test recording, the LLM has almost no real content to work with but is instructed by an extremely detailed system prompt (with NHS-specific section templates, topic clusters, governance formats) to produce 1,500+ words of structured minutes. The model fills the gap with plausible-sounding fabrications drawn from the prompt's own examples (PCN DES, ARRS, Bugbrooke, etc.).

2. **The overview generator also hallucinates.** The `generate-meeting-overview` function uses the hallucinated notes as input, compounding the fabrication into the meeting card summary visible on the home page.

## Plan

### 1. Add server-side short transcript guard in `generate-meeting-notes-claude`

After line 588 (`if (!transcript) throw new Error('Transcript is required');`), add a word count check:

- Count words in the transcript
- If fewer than 100 words, return a standardised "insufficient content" response instead of generating notes
- The response should contain a brief factual summary: state that the recording captured minimal content, list the approximate word count, and note that no substantive meeting content was identified
- This prevents the LLM from ever seeing an ultra-short transcript with the full governance prompt

### 2. Add transcript length-aware prompting for short-but-valid transcripts

For transcripts between 100-300 words:

- Prepend an instruction to the system prompt: "This is a very short transcript. Report ONLY what is explicitly stated. Use 'None identified' for all sections where the transcript provides no information. Do NOT extrapolate, infer, or generate plausible content."
- Reduce `max_completion_tokens` to 2000 to limit output length
- This handles edge cases where a short meeting is legitimate but shouldn't trigger full governance-grade output

### 3. Add the same guard to `generate-meeting-overview`

In the overview function, before calling the AI:

- Count words in the content being summarised
- If fewer than 100 words, save a simple factual overview: "This recording captured minimal content (X words). No substantive meeting topics were identified."
- Skip the AI call entirely for ultra-short content

### 4. Add footer disclaimer on short-transcript notes

Append `\n\n---\n\n*Generated by Notewell AI — MHRA Class I registered medical device. These minutes are based solely on the content of the provided transcript. No information has been inferred, estimated, or supplemented beyond what was explicitly recorded.*` — this already exists in the prompt but should be enforced as a post-processing step for all notes, especially short ones.

## Files to Change

- `supabase/functions/generate-meeting-notes-claude/index.ts` — add word count guard + short-transcript prompt modifier
- `supabase/functions/generate-meeting-overview/index.ts` — add word count guard
