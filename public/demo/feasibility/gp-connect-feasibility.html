§ 1

Executive summary

GP Connect is the correct architectural choice for pushing structured clinical tasks from Notewell AI into EMIS Web and TPP SystmOne. The capability exists today, runs on production NHS infrastructure, and is already in daily live use across England.

The engineering effort to build the integration is modest — estimated at four to six weeks of focused development. The dominant constraint is not technology; it is the NHS Digital assurance and onboarding process, and the willingness of individual GP practices to enable the receiving endpoint.

Bottom line for DDaT

There is no technical blocker. The NHS has already built the rails — Spine, MESH, GP Connect APIs, FHIR UK Core — and they are production-grade and in widespread use.

What we are proposing is not novel integration. It is the same pattern used by every hospital eDischarge system, every 111 call-centre platform, and every accredited community correspondence tool already in operation.

The question is not "can this be done?" — it is "when do we start, and which practice pilots it first?"

§ 2

Context

Current state

The AgeWell home frailty review produces a comprehensive Patient Support Plan and a set of agreed actions. Of those actions, a subset require GP review: medication reviews, postural hypotension workups, mood screening follow-ups, immunisation confirmations, inhaler technique checks.

Today those GP-bound actions leave the AgeWell workflow as free-text paragraphs in a document that must be emailed, faxed or — increasingly rarely — hand-delivered. They are then re-keyed by a practice administrator into a task in the GP clinical system, with lossy translation and variable timeliness. The loop is not closed.

Target state

When the AgeWell Support Worker finalises a patient's support plan in Notewell, the GP-bound actions flow electronically and structurally into the registered GP's task inbox within minutes. Each action arrives as a coded, structured, patient-matched task that the GP can action directly in SystmOne or EMIS Web, exactly as they would an eDischarge item or a pathology result.

§ 3

GP Connect capability assessment

GP Connect is not a single API; it is a family of capabilities published by NHS England. Each has a different maturity, fit, and assurance bar. The table below assesses each against the AgeWell requirement.

CapabilityFHIRPurposeMaturityFit
Access Record: HTMLSTU3Read-only view of patient record as rendered HTMLLiveRead only
Access Record: StructuredSTU3 / R4Programmatic FHIR retrieval of record sectionsLiveRead only
Appointment ManagementSTU3Book / view / cancel appointments at a GP practiceLivePartial
Send DocumentR4 (UK Core)Push document + structured coded content into GP workflow inboxLivePrimary
Update RecordR4 (UK Core)Write coded clinical content directly into the structured recordEmergingPhase 3

Why Send Document is the right starting point

  • Proven pattern. Used in production for hospital eDischarge, 111 post-event messaging, and mental-health correspondence. Handled as a first-class workflow item by both TPP and EMIS.
  • Appears in the right place. Tasks arrive in the practice's native workflow manager — no new interface for the GP to learn.
  • Structured AND narrative. Carries both a readable document and FHIR Task resources. Satisfies clinical safety and interoperability requirements simultaneously.
  • Lowest assurance bar of the "write" capabilities. Easier and faster to clear than Update Record, which requires coded terminology governance beyond what is proportionate today.

Why MESH, not direct FHIR REST?

GP Connect supports both transports. For SendDocument the national recommendation is MESH because it decouples sender uptime from receiver availability, is designed for store-and-forward with delivery guarantees, and is the existing rail all practice systems already poll. It also means we don't need to expose a public inbound endpoint to Spine.

§ 5

FHIR payload structure

The payload sent to MESH is a FHIR R4 Bundle of type document, conforming to the UK Core profile set and the GP Connect SendDocument StructureDefinition. It contains one Composition, one Patient, one Practitioner, one Organization, and one Task resource per GP-bound action item.

{
  "resourceType": "Bundle",
  "type": "document",
  "identifier": { "system": "https://notewell.co.uk/bundle", "value": "NW-20260415-472" },
  "timestamp": "2026-04-15T16:42:00+01:00",
  "entry": [

    // --- 1. The Composition: metadata + narrative ---
    { "resource": {
        "resourceType": "Composition",
        "status": "final",
        "type": { "coding": [{
          "system":  "http://snomed.info/sct",
          "code":    "861421000000109",
          "display": "Care plan report"
        }]},
        "subject": { "reference": "Patient/dot-pearson" },
        "author":  [{ "reference": "Practitioner/sarah-mitchell" }],
        "title":   "AgeWell Patient Support Plan — Mrs D Pearson",
        "section": [{
          "title": "GP-requested actions",
          "entry": [
            { "reference": "Task/task-1" },
            { "reference": "Task/task-2" },
            /* ... 3 more tasks ... */
          ]
        }]
    }},

    // --- 2. The Patient (NHS number is the key match) ---
    { "resource": {
        "resourceType": "Patient",
        "identifier": [{
          "system": "https://fhir.nhs.uk/Id/nhs-number",
          "value":  "4382917654"
        }],
        "name":      [{ "family": "Pearson", "given": ["Dorothy","Margaret"] }],
        "gender":    "female",
        "birthDate": "1942-03-15"
    }},

    // --- 3. One Task per requested action ---
    { "resource": {
        "resourceType": "Task",
        "status":    "requested",
        "intent":    "order",
        "priority":  "urgent",
        "description": "Postural hypotension workup",
        "for":       { "reference": "Patient/dot-pearson" },
        "owner":     { "reference": "Practitioner/dr-a-patel" },
        "code": { "coding": [{
          "system":  "http://snomed.info/sct",
          "code":    "79491001",
          "display": "Postural hypotension evaluation"
        }]},
        "note": [{ "text":
          "Sit→stand drop 14 mmHg. Review antihypertensives. Repeat L/S BP at 2/52."
        }]
    }}

}
import { v4 as uuid } from "uuid";
import type { SupportPlan, ActionItem } from "@/types/supportPlan";

export function composeSendDocumentBundle(plan: SupportPlan) {
  const now = new Date().toISOString();
  const patientRef = `urn:uuid:${uuid()}`;
  const authorRef  = `urn:uuid:${uuid()}`;

  const tasks = plan.actions
    .filter(a => a.destination === "GP")
    .map(a => makeTask(a, patientRef, authorRef));

  return {
    resourceType: "Bundle",
    type:         "document",
    identifier:   { system: "https://notewell.co.uk/bundle", value: plan.id },
    timestamp:    now,
    entry: [
      { fullUrl: "urn:uuid:composition", resource: makeComposition(plan, patientRef, authorRef, tasks) },
      { fullUrl: patientRef,             resource: makePatient(plan.patient) },
      { fullUrl: authorRef,              resource: makePractitioner(plan.worker) },
      ...tasks.map(t => ({ fullUrl: `urn:uuid:${t.id}`, resource: t })),
    ],
  };
}

function makeTask(action: ActionItem, patientRef: string, authorRef: string) {
  return {
    resourceType: "Task",
    id:           `task-${action.id}`,
    status:       "requested",
    intent:       "order",
    priority:     action.priority,            // routine | urgent | asap
    description:  action.title,
    for:          { reference: patientRef },
    requester:    { reference: authorRef },
    code: {
      coding: [{
        system:  "http://snomed.info/sct",
        code:    action.snomedCode,
        display: action.snomedDisplay,
      }],
    },
    note: [{ text: action.clinicalDetail }],
  };
}

§ 6

Authentication, transport and security

The JWT handshake

Every GP Connect call carries a signed JSON Web Token identifying the sending and receiving organisations. This is the mechanism Spine uses to establish trust. Signing key is an RSA private key issued to PCN Services Ltd as part of onboarding; public key published at a well-known JWKS endpoint for Spine to verify.

// JWT payload (decoded) — every outbound GP Connect message carries this
{
  "iss":    "https://notewell.co.uk",                           // Notewell as issuer
  "sub":    "https://fhir.nhs.uk/Id/sds-user-id|<worker-sds>",
  "aud":    "https://PSIS/STU3/1/gpconnect/documents/fhir",
  "exp":    1776492000,                                          // 5-min expiry
  "iat":    1776491700,
  "reason_for_request": "directcare",
  "requested_scope":    "patient/*.write",
  "requesting_organization": {
    "identifier": [{
      "system": "https://fhir.nhs.uk/Id/ods-organization-code",
      "value":  "X26ABC"                                    // Notewell/PCN Services Ltd ODS
    }]
  },
  "requesting_practitioner": {
    "identifier": [{
      "system": "https://fhir.nhs.uk/Id/sds-user-id",
      "value":  "<Sarah Mitchell SDS user ID>"
    }]
  }
}

§ 7

NHS Digital onboarding and assurance

This is the longest-duration workstream and the one most often underestimated. The technical build can run in parallel, but neither reaches production without the other.

The 8-stage assurance timeline. Two of eight stages already in place (DSPT, DCB0129); the remaining six form the critical path.
Stages already complete or substantially complete

MHRA Class I medical device registration · DCB0129 Clinical Risk Management with Dr Simon Ellis as CSO · DSPT Standards Met · ICO registration (ZB226324) · DTAC partially complete

This is not a greenfield assurance exercise. Notewell's existing governance posture shortens the full timeline by an estimated 3 months compared to a new supplier starting from zero.

Practice-side enablement

Each receiving practice must enable their end of the pipe. This is typically a 30-minute task for their GP system vendor but must be scheduled.

  • SystmOne: TPP enables the GP Connect provider endpoint for the practice and confirms the MESH mailbox. Standard contract — no charge.
  • EMIS Web: Enabled via EMIS Application Services Management (ASM). Standard service for accredited senders — no charge.
  • Data Sharing Agreement: Extension of the existing Notewell practice agreement; DPIA already auto-generated at onboarding.

Engineering cost summary

~9 weeks development + ~4 weeks testing running in parallel for most streams. One senior full-stack engineer with existing Notewell / TypeScript / Supabase experience can deliver this. A second engineer parallelising FHIR-composer and MESH-client streams would compress elapsed time to ~5 weeks.

Specialist Spine / MESH integration consultancy (~5 days at ~£1,200/day) is recommended insurance for the authentication work. Indicative external spend: £6–10k.

§ 9

Testing strategy

Test pyramid: broad coverage at the base with unit and schema tests; narrower but higher-value tests at the top, culminating in real-practice pilot UAT.

Pilot success criteria

  • 100% of GP-bound action items arrive in the target practice's workflow manager within 5 minutes of plan finalisation.
  • Zero messages lost, zero misrouted to wrong patient, zero PDS mismatches.
  • GP completion in SystmOne/EMIS flows back to Notewell within 30 minutes.
  • No increase in clinical safety incidents over the 4-week pilot vs baseline.
  • GP qualitative feedback: median ≥ 4/5 for "does this save you time".

Risks and mitigations

Practice declines to enable receiver endpoint Low Medium Both TPP and EMIS support GP Connect as standard. Towcester MC nominated — Dr Patel is clinical champion.
FHIR Bundle rejected with obscure validation error Medium Low Pre-send CI validation against latest UK Core StructureDefinitions. Reference implementation open-source and runs locally.

MESH mailbox saturation or delay Low Medium Volume <50 msg/day even at scale. Dashboard alerts on msg age >15 min. NHSmail fallback documented.
GP assumes "AI has handled it" Low High Task arrives like an eDischarge — no AI handoff inference. Clinician remains responsible. Covered in DCB0160.
Scope creep to Update Record prematurely Medium Medium Phase lock: Update Record deferred to 2027 in programme governance document.
§ 11

Phased delivery

The risk of a direct GP Connect bet is a long tail of clinical benefit waiting on assurance. The recommended approach is three phases that de-risk each step and deliver user-visible value at every stage.

Phase 1
NHSmail structured delivery
1 week build Live in days
Auto-generated structured NHSmail with PDF + JSON sidecar to practice shared inbox. Not structured at receipt, but low-friction — practices already process NHSmail daily. 70% of the benefit at 10% of the cost.
Phase 2
GP Connect Send Document
9 wks build + 4 wks test 6–9 months
Structured FHIR payload; tasks appear natively in SystmOne/EMIS workflow; ACK loop closes. This paper's central recommendation and the clinical-benefit anchor.
Phase 3
GP Connect Update Record
12 wks build + 8 wks test 12–18 months
AgeWell observations, BP readings, PHQ-9 scores, Rockwood CFS written directly into the structured GP record as coded entries. Longer-term aspiration; deferred to 2027.

Recommendations for DDaT

Three decisions are requested.

1
Endorse the three-phase approach
Phase 1 NHSmail MVP in the next sprint. Phase 2 GP Connect Send Document over 6–9 months. Phase 3 Update Record explicitly deferred to 2027. This phased envelope is what delivers user-visible value quickly while de-risking the assurance tail.
2
Authorise GP Connect onboarding submission
Authorise PCN Services Ltd to file the onboarding request with NHS England. No commercial commitment at this stage; the request simply initiates the assessment process and starts the 4–5 month clock that otherwise sits idle.
3
Nominate Towcester Medical Centre as Phase 2 pilot
Dr Patel and the Towcester MC team are the operational anchor for AgeWell locally. Nominating them as the Phase 2 pilot aligns engineering, clinical, and operational risk in a single location with strong existing relationships.
Closing note

This integration is not speculative. It uses NHS infrastructure that 7,000+ English GP practices already rely on daily. The work required is disciplined engineering plus a well-understood assurance process — not invention.

We are proposing to close a documentation loop today closed manually by administrative staff. The outcome for patients is timelier clinical action. The outcome for clinicians is less re-keying. The outcome for the ICB is better data, stronger interoperability posture, and a reference implementation of primary-care-originated GP Connect messaging that can be extended across neighbourhood services.
